When a Key Vault certificate renews against the wrong issuer, the consequences go beyond an availability disruption. The portal showed browser trust errors because no public CA validated the identity behind the certificate—patients had no way to verify they were connecting to Contoso Health rather than an attacker presenting an identical-looking self-signed cert. Users conditioned to quickly pass through the warning become indistinguishable targets for a man-in-the-middle substitution. And unlike CA-issued certificates, self-signed certs have no revocation infrastructure—if the private key is ever compromised, there's no standard mechanism to invalidate it. The misconfiguration was a single wrong value in a certificate policy. Understanding how Key Vault manages certificate issuance is what prevents it from happening again.

## Distinguish certificate issuance models

Azure Key Vault supports two certificate issuance models.

**Self-signed certificates** are generated by the vault itself—Key Vault acts as the CA. Only systems that explicitly import it as a trusted root trusts the certificate. They're appropriate for internal service-to-service communication in controlled environments and nonproduction testing. They're inappropriate for public-facing endpoints where browsers, operating systems, and Transport Layer Security (TLS) clients validate against a trusted CA chain.

**CA-integrated certificates** are issued by a trusted certificate authority with a public root in browser trust stores. Azure Key Vault has native partnerships with **DigiCert** and **GlobalSign** for automated, end-to-end certificate lifecycle management. When you configure Key Vault to use an integrated certificate authority (CA), the vault handles the entire workflow: it generates the private key, submits the certificate signing request (CSR) to the CA, receives the signed certificate, and stores it—all without you interacting with the private key material. The workflow is fully managed.

## Configure an integrated CA issuer

Before Key Vault can issue or renew certificates through DigiCert or GlobalSign, you configure an **issuer object** in the vault. This object stores the CA account credentials and contract details:

:::image type="content" source="../media/key-vault-certificate-authorities-portal.png" alt-text="Screenshot of the Azure portal showing the Certificate Authorities pane inside a Key Vault resource with the Add button highlighted.":::

1. In the Azure portal, navigate to your key vault and select **Certificates** > **Certificate Authorities**.
2. Select **Add** and choose the provider (DigiCert or GlobalSign).
3. Enter your CA account ID, API key or password, and any organization details required by the provider.
4. Save the issuer. The vault now has a named issuer object (for example, `contoso-digicert-issuer`) that certificate policies can reference.

Any certificate policy that references this issuer submits renewal requests to DigiCert or GlobalSign automatically when the lifetime action triggers. The CA receives the CSR, validates the domain and organization, issues the signed certificate, and Key Vault stores it. No manual workflow, no opportunity for a misconfigured self-signed fallback.

> [!NOTE]
> Nonpartnered providers (internal enterprise CAs, external CAs without a Key Vault integration) are also supported, but they don't support automatic renewal. For those CAs, you import the certificate manually and manage renewal outside of Key Vault or through custom automation.

## Configure lifetime actions for autorenewal

A **certificate policy** in Key Vault defines both what the certificate looks like and the certificate management over time. The lifecycle section of the policy specifies **lifetime actions—what happens as the certificate approaches expiry.

You configure the trigger as one of:

- A **percentage of lifetime** remaining (for example, autorenew when 80% of the certificate's lifetime elapses). Percent lifetime is relative and works well for certificates with varying validity periods.
- **Days before expiry** (for example, autorenew 30 days before expiry). A set number of days is absolute and predictable.

The action for CA-integrated certificates is `autoRenew`. When the trigger fires, Key Vault generates a new CSR using the same policy and submits it to the configured issuer. When the CA returns the signed certificate, Key Vault stores it as a new version.

:::image type="content" source="../media/certificate-auto-renewal-workflow.png" alt-text="Flowchart showing the autorenewal sequence from lifetime trigger through CA issuance to Key Vault storing the new certificate version." lightbox="../media/certificate-auto-renewal-workflow.png":::

For certificates backed by an external or nonpartnered CA, the action is `emailContacts`. This triggers a notification to the designated **certificate contacts** so the renewal can be handled manually.

## Set certificate contacts

Certificate contacts are email addresses or distribution lists registered at the vault level—the same contacts apply to every certificate in the vault, not to individual certificates. They're notified when:

- A certificate is nearing expiry.
- An autorenewal attempt succeeded.
- An autorenewal attempt failed.

For critical certificates, designate a team distribution list as the contact—not an individual email. Individual contacts become a single point of failure for renewals. A team contact ensures that certificate lifecycle events reach the right people regardless of individual availability.

## Evaluate certificate exportability

A certificate's exportability is controlled by the `exportable` key property in its certificate policy. When set to nonexportable, the private key lives in the vault and is never included in the secret representation that callers retrieve. Applications that need to present the certificate—TLS endpoints, mutual TLS authentication—access it through Key Vault's API at operation time.

**Exportable certificates** allow the private key to be retrieved as a PFX or PEM file, which is required for workloads that need the certificate injected into a virtual machine, container, or platform that can't natively call the Key Vault API. Retrieving the private key requires access to the certificate's secret representation—Key Vault stores the private key as an addressable secret with the same name as the certificate. The **Key Vault Secrets User** role grants this access; the **Key Vault Certificate User** role grants access to the full certificate including its private key portion.

When you configure exportable certificates, restrict the access policies carefully. Anyone with the ability to export an exportable certificate can extract the private key from the vault boundary—which largely negates the key material protection that Key Vault provides.

> [!IMPORTANT]
> HSM-backed keys are nonexportable by design. If you need key material portability (export), use a software-backed RSA or EC key for the certificate. The decision between HSM protection and export capability is mutually exclusive for RSA and EC certificates.
