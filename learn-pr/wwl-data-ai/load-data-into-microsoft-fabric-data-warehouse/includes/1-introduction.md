A data warehouse in Microsoft Fabric stores business data in an optimized format for analytics, but it's only valuable when you can efficiently load data from your source systems. Choosing the right loading strategy determines how quickly you can refresh your analytics and how much transformation logic you can apply during ingestion.

Imagine you're a data engineer at a retail company that's migrating to Microsoft Fabric. Your sales data lives in Azure Blob Storage as daily CSV exports, your product catalog updates come from an on-premises SQL Server database, and your customer activity logs stream into OneLake from various web applications. Your analytics team needs this data consolidated in a warehouse for Power BI reports that refresh every morning. You need a loading solution that handles both one-time historical loads and daily incremental updates, and you want the flexibility to apply transformations like slowly changing dimensions and surrogate key generation. Some data sources require simple file copies, while others need complex data type conversions and business rule validation before landing in the warehouse tables.

In this module, you explore three common approaches for loading data into a Fabric warehouse: Copy jobs in data pipelines for wizard-driven ingestion, T-SQL statements like COPY and CREATE TABLE AS SELECT for code-based loading, and Dataflow Gen2 for visual Power Query transformations. Fabric also supports loading through notebooks, REST APIs, and other methods. You also examine data load strategies including full versus incremental loads, slowly changing dimension patterns, and surrogate key management.

By the end of this module, you're able to select the appropriate loading method based on your data sources, transformation requirements, and refresh frequency.
