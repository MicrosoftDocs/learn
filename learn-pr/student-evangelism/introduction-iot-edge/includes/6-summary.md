Consider the example of a doctor who needs to respond immediately when a patient's glucose levels change. The doctor uses an IoT device to monitor those levels, and an algorithm running on the device creates the alert. The device runs even when connectivity is poor. In this situation, supported IoT Edge can help solve the doctor's problem by running Linux container modules on supported Linux hosts, or on supported Windows hosts through IoT Edge for Linux on Windows (EFLOW). IoT Edge can help run modules (algorithms) on edge devices under intermittent connectivity after the IoT Edge device has connected to IoT Hub and synced at least once. Queued upstream messages are stored and forwarded on reconnect, subject to configured time-to-live settings and available storage. Security still requires solution-specific design, such as certificate-based authentication, storage permissions, persistent host storage, and encryption at rest. Without IoT Edge, the developer would have to build all of this functionality themselves (for example, security, machine learning deployment, and offline operation). With IoT Edge, you can focus on the business problem at hand (providing care to the patient and responding to emergency scenarios).

Therefore, a patient-response system must operate within a defined set of performance criteria. The system also has to account for component limitations (such as cloud response times) and mitigate them with techniques such as edge computing, as well as physical constraints (such as low or absent connectivity) and secure data management on devices.
