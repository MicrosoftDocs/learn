In this section, we'll discuss how you can decide whether IoT Hub is the right choice for your solution. We'll list some of the criteria that you can use to create IoT solutions in the cloud that are capable of scaling. Many cloud-connected Azure IoT solutions use IoT Hub as a central message hub between devices and back-end services. (Other Azure IoT options exist for specific scenarios — for example, Azure IoT Operations for industrial/edge deployments and Azure Event Grid's MQTT broker for lightweight pub/sub messaging — but IoT Hub remains the most general-purpose choice for connecting and managing devices at scale.) The exact configuration of IoT Hub you should use depends on the selection criteria below:

## Decision criteria

The primary considerations for selection include:

|  | Description |
|:---:|---|
|                  **Application complexity**                  | Azure IoT Hub offers three tiers: Basic (B1, B2, B3), Standard (S1, S2, S3), and Free (F1) for evaluation and testing. The Basic tier supports one-way (device-to-cloud) telemetry and device file upload, plus per-device identity, message routing and message enrichments, Event Grid integration, support for HTTPS, AMQP, and MQTT protocols, and monitoring and diagnostics. The Standard tier adds the features Basic doesn't support: cloud-to-device messaging, device twins, module twins, device management, Azure IoT Edge, and IoT Plug and Play. The Azure IoT Hub Device Provisioning Service (DPS) is a separate companion service that integrates with both Basic and Standard tier IoT Hubs to register devices automatically at scale.<br><br>Choose the Standard tier when your scenario requires bidirectional device communication, configuration management through device twins, IoT Edge workloads, or Plug and Play model-based interactions. Choose the Basic tier for telemetry ingestion and device file-upload scenarios that don't need cloud-to-device communication or device-management features. Use the Free tier only for evaluation, demos, or learning. |
|                     **Data throughput**                      | Data throughput depends on both the amount of data you move through IoT Hub and the rate of key operations. Basic and Standard tiers are available in sizes 1, 2, and 3; the Free tier is F1 only and is intended for evaluation and testing. Each IoT hub is provisioned with a tier, size, and number of units that determine its daily message quota. When estimating daily usage, use the metered message size for the tier (0.5 KB for Free and 4 KB for Basic and Standard), so larger payloads can consume multiple metered messages. Also check the expected device-to-cloud send rate, new device connection rate, and operation throttles such as cloud-to-device sends, direct methods, twin operations, jobs, file uploads, and queries. Each unit of a size 1 IoT hub can handle 400,000 messages per day, each unit of a size 2 hub can handle 6 million messages per day, and each unit of a size 3 hub can handle 300 million messages per day. Scale up by changing edition size within a tier (for example, B1→B2/B3 or S1→S2/S3), and scale out by increasing units in the selected edition; adding units can increase daily quota and many throttles within documented limits. |
| **Securing the solution end-to-end with per-device authentication** | IoT Hub authenticates each device individually using either an X.509 certificate (recommended for production) or a symmetric-key SAS token. For back-end (service-side) access to IoT Hub APIs, Microsoft Entra ID with Azure built-in roles (such as IoT Hub Data Contributor and IoT Hub Data Reader) is the recommended mechanism; shared-access-policy keys remain available for compatibility but can be disabled. Don't disable shared-access-policy authentication when you rely on DPS provisioning flows unless you have planned the migration or an alternative, because DPS links IoT hubs using shared access policies and provisioning can fail if that access is denied. |
|               **Bidirectional communication**                | Azure IoT Hub can be used to establish bidirectional communication with millions of simultaneously connected IoT devices. (A single IoT Hub instance has a documented limit of 1,000,000 total device and module identities; for very large fleets, multiple IoT Hub instances can be coordinated through the Device Provisioning Service.) You can use device-to-cloud telemetry data to understand the state of your devices and define message routes to other Azure services without writing any code. For cloud-to-device communication, choose the option that matches the interaction pattern: direct methods for immediate commands that need a prompt response, device twin desired properties for desired state or long-running configuration changes, and cloud-to-device messages for one-way notifications. Cloud-to-device messages can be acknowledged and retained for offline devices until their time-to-live expires. Their delivery count increases only when a device explicitly abandons a message; if the message lock expires, IoT Hub returns the message to the queue without incrementing the delivery count. |

|  | Question |
|:---:|---|
|         **Telemetry**         | Does your solution need only basic telemetry services?       |
|    **Geographic coverage**    | Does your solution need comprehensive geographic coverage?   |
|    **Support for devices**    | Do you need to support a range of devices for your solution? |
| **Manage a range of devices** | Do you need to track and manage data for different kinds of devices? |
|  **Communication protocols**  | Does your solution need to connect over different kinds of communication protocols? |
|      **Message routing**      | How do you forward telemetry from particular devices to the appropriate back-end services (for example, route a hot path to Event Hubs or Service Bus for real-time processing, and a cold path to Azure Cosmos DB or Azure Storage for long-term analytics)? IoT Hub message routing applies query-based filters to incoming device-to-cloud messages and sends each message to every endpoint whose route conditions it matches, including the built-in Event Hubs-compatible endpoint or configured custom endpoints such as Storage containers, Event Hubs, Service Bus queues/topics, or Cosmos DB. If multiple matching routes point to the same endpoint, IoT Hub delivers the message to that endpoint only once. After you create any custom route, device-to-cloud data no longer flows automatically to the built-in endpoint; add a route that targets the built-in endpoint if consumers must receive matching telemetry there, or enable the fallback route to send only messages that don't match any route to the built-in endpoint. Azure Functions can process IoT Hub events from the built-in/Event Hubs-compatible endpoint, from a routed Event Hubs endpoint with an Event Hubs trigger, or through IoT Hub's Event Grid integration. |
