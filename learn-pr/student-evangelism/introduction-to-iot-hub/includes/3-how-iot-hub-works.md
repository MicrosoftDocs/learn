Azure IoT Hub is a managed Azure PaaS (Platform as a Service) that enables bidirectional communications between IoT devices and a cloud-based IoT solution. It implements essential functions that are common to many cloud-based IoT deployments, including secure connectivity, device-to-cloud and cloud-to-device messaging, per-device identity, message routing, monitoring, and integration with downstream Azure services for compute (for example, Azure Functions and Azure Stream Analytics) and storage (for example, Azure Storage and Azure Cosmos DB). IoT Hub isn't the starting point for every IoT solution: edge-based or hybrid deployments, managed application scenarios, and MQTT broker-style publish/subscribe scenarios might use Azure IoT Operations, Azure IoT Central, Azure Event Grid MQTT broker, or other Azure services. We describe the IoT Hub features below.

**Protocols supported:** IoT Hub allows devices to use the following protocols for device-side communications: MQTT, MQTT over WebSockets, AMQP, AMQP over WebSockets, and HTTPS. Plan devices and service clients to use TLS 1.2 and supported strong cipher suites. Protocol choice affects feature support: HTTPS is useful for devices that can't support other protocols, but it doesn't support device twin operations or receiving direct method requests, and devices that use HTTPS must poll IoT Hub for cloud-to-device messages. MQTT and AMQP support server push for cloud-to-device messages. IoT Hub provides limited MQTT feature support; use Azure Event Grid MQTT broker when your solution needs MQTT v5 or richer broker-style publish/subscribe features beyond IoT Hub's MQTT support.

**Device Identity Registry**: IoT Hub maintains an identity registry. The identity registry stores information about the devices and modules permitted to connect to the IoT Hub. An identity registry entry for the device or module must exist before the entity can connect to the IoT Hub. A device or module must also authenticate with the IoT Hub based on credentials stored in the identity registry. 

**Authentication**: IoT Hub supports separate authentication paths for devices and service-side applications. For device authentication, IoT Hub supports symmetric-key authentication with Shared Access Signature (SAS) tokens and X.509 certificate authentication.

With symmetric-key authentication, IoT Hub stores primary and secondary keys in the identity registry for each registered device or module. The device authenticates by presenting a time-limited SAS token derived from one of those keys, signed for a specific resource URI and expiry. With X.509 certificate authentication, the device authenticates during the TLS connection by using either a CA-signed certificate that chains to a Certificate Authority (CA) registered with IoT Hub, or a self-signed certificate registered to the device by thumbprint. CA-signed certificates are recommended for production deployments because they scale from a single root of trust; self-signed certificates are typically used for testing or smaller deployments.

For service-side (back-end) access to IoT Hub service APIs — for example, an application that manages device identities, reads or updates twins, invokes direct methods, or sends cloud-to-device messages — Microsoft recommends authenticating with Microsoft Entra ID and authorizing access via Azure role-based access control (RBAC) using built-in roles such as IoT Hub Data Contributor, IoT Hub Data Reader, IoT Hub Twin Contributor, and IoT Hub Registry Contributor. Microsoft Entra ID and RBAC apply to IoT Hub service APIs, not to IoT Hub device APIs. Reading telemetry from the built-in Event Hubs-compatible endpoint (**messages/events**) is also an exception: that endpoint still uses an Event Hubs-compatible connection string with shared access keys. If your solution requires Microsoft Entra-based telemetry consumption, route device-to-cloud messages to a custom Event Hubs endpoint and consume from that endpoint with Microsoft Entra ID and RBAC. Shared-access-policy authentication is still supported for compatibility, but if you plan to disable it, first confirm service clients use Microsoft Entra/RBAC and that you don't rely on Device Provisioning Service (DPS) flows that link IoT hubs by using shared access policies. Device-side authentication continues to use X.509 certificates or SAS tokens as described above.

**Device twins**: Device twins are JSON documents that store device state information, including metadata, configurations, and conditions. Azure IoT Hub implicitly creates a device twin when the related device identity is created, and deletes the twin when that identity is deleted. Device twins store device-related information that devices and back-end applications can use to synchronize device conditions and configuration. Each device twin contains tags (back-end metadata about the device), desired properties (target state set by back-end services), and reported properties (current state reported by the device).

**Endpoints that IoT Hub exposes:** For each device in the identity registry, IoT Hub exposes a set of endpoints: Send device-to-cloud messages; Receive cloud-to-device messages; Initiate file uploads; Retrieve and update device twin properties (not supported over HTTPS); Receive direct method requests (not supported over HTTPS). IoT Hub currently supports the following Azure services as additional (custom) endpoints for message routing: Azure Storage containers (Blob storage / ADLS Gen2), Event Hubs, Service Bus queues, Service Bus topics, and Azure Cosmos DB. With message routing, a device-to-cloud message is delivered to every endpoint whose route query it matches; if multiple routes target the same endpoint, IoT Hub delivers the message to that endpoint only once. Once you create any route, messages stop flowing to the built-in **messages/events** endpoint unless you create a route to that endpoint or enable the fallback route. The fallback route sends only messages that don't match any other route, so create an explicit route to the built-in endpoint if consumers need all routed telemetry there.

**Tier note:** The Basic tier is intended for device-to-cloud telemetry scenarios. Use the Standard tier when your solution needs device twins, module twins, cloud-to-device communication, direct methods, IoT Edge, or device-management features. The Free tier includes Standard-tier capabilities for evaluation, with limited messaging allowances.

**Provisioning devices with Azure IoT Hub Device Provisioning Service:** The IoT Hub Device Provisioning Service is a helper service for IoT Hub. The device provisioning service enables zero-touch, just-in-time provisioning to the right IoT Hub without requiring human intervention, allowing the customers to provision millions of devices in a secure and scalable manner.

**The telemetry function** is an essential component of IoT Hub. The telemetry function involves recording and transmitting values received by an IoT device. However, IoT Hub is much more than the basic telemetry function. The **scaling feature** of IoT Hub allows you to ramp up (or down) the scope of the solution. The ability to scale a solution depends on two considerations: the features you plan to use and the amount of data you plan to move daily. Once you plan to deploy devices at scale, you need to manage these devices. IoT Hub maintains device and module identities and supports device-management capabilities, while at-scale zero-touch provisioning is handled by Device Provisioning Service (DPS) used with IoT Hub. DPS assigns devices to the right IoT hub and registers them there so they can connect directly and authenticate. The security requirements can be seen as part of a **security function**, which manages the per-device authentication and access requirements with multiple authentication types. Based on the security functions, the **routing function** determines the message flow and the recipients of the message. Finally, you can connect to devices natively using the **Azure IoT device SDKs**, and integrate with other Azure services through built-in **service integrations** such as message routing, Event Grid integration, and the Azure IoT Hub REST and management APIs.

If you consider the scenario outlined before about solution developers working with multiple sensors in the city – the prototype would have started with a basic telemetry function. But to deploy citywide – many more elements are needed. These include the ability to use DPS for at-scale provisioning, manage device identities and lifecycle in IoT Hub, implement device-level security, and ensure message-level routing. Due to the diversity of devices in a city, the ability to connect natively using an SDK will expand the user base of the solution. In the absence of the functionality from IoT Hub and related Azure IoT services, the solution would be hard to deploy beyond a basic telemetry function.
