When you sign in to a Windows 365 Cloud PC, you're not starting from a blank machine, you're entering a development environment that's already been prepared. This flowchart illustrates key ways the two approaches differ:

### From blank machine to ready-to-code Cloud PC

:::image type="content" border="true" source="../media/comparison.svg" alt-text="Diagram that shows a traditional developer setup and a Windows 365 Cloud PC, highlighting fewer steps and faster coding.":::

In traditional setups, developers configure each environment individually. With Windows 365, environments are prepared centrally and delivered to developers, reducing setup effort and improving consistency.

This approach changes how development environments are delivered. It can reduce onboarding friction, help teams maintain consistent environments at scale, and lower the effort required to support developer workstations.

As a result, you can:

- Begin development work shortly after signing in, without completing many typical setup steps
- Work across Windows and Linux workflows within the same environment
- Move between devices without reconfiguring
- Rely on a consistent experience across teams and projects

> [!NOTE]
> While Windows 365 offers several benefits for onboarding developers quickly, a fully local workstation might be preferred when online connectivity or data use is limited.
